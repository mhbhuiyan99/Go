# Beego vs Raw Go `net/http` — A Practical Guide

> **Who this is for:** Go developers who are comfortable with the standard library
> (`net/http`) and want to understand how Beego works and why you'd use it.

---

## What is Beego?

Beego is a full-featured **MVC web framework** for Go. It is built on top of Go's
standard `net/http` package — meaning it doesn't replace the standard library, it
wraps and extends it to reduce boilerplate.

It provides:
- A structured way to organize handlers (Controllers)
- Automatic HTTP method dispatch (`GET`, `POST`, `PUT`, `DELETE`)
- A namespace-based router
- A configuration system (`app.conf`)
- Built-in logging
- JSON response helpers

**Official package:** `github.com/beego/beego/v2`

---

## The Core Mental Shift

| Raw `net/http` | Beego |
|---|---|
| You think in **functions** | You think in **structs with methods** |
| One function handles one route | One struct handles one resource |
| HTTP method checked manually | HTTP method dispatched automatically |

## Side-by-Side Comparison

To make the differences concrete, let's build the same thing both ways:
a simple `GET /api/v1/health` endpoint that returns a JSON response.

---

### Raw `net/http`

```go
package main

import (
    "encoding/json"
    "net/http"
)

func healthHandler(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(map[string]interface{}{
        "success": true,
        "message": "Server is running",
    })
}

func main() {
    http.HandleFunc("/api/v1/health", healthHandler)
    http.ListenAndServe(":8080", nil)
}
```

You manually handle everything: setting headers, writing status codes,
encoding JSON, and starting the server.

---

### Beego

In Beego, the same endpoint is split across three concerns:
the base controller, the handler, and the router.

**Step 1 — Define a BaseController**

`BaseController` is a struct that embeds Beego's `web.Controller`.
This is where you put shared helper methods that every controller in your
application can reuse. This is a **standard Beego pattern** used in virtually
every Beego project — not specific to any one project type.

```go
// controllers/base.go
package controllers

import "github.com/beego/beego/v2/server/web"

type BaseController struct {
    web.Controller
}

// RespondJSON is a shared helper all controllers inherit.
// It sets the HTTP status, builds a consistent response shape,
// and writes the JSON response.
func (c *BaseController) RespondJSON(status int, success bool, message string, data interface{}) {
    c.Ctx.Output.SetStatus(status)
    c.Data["json"] = map[string]interface{}{
        "success": success,
        "message": message,
        "data":    data,
    }
    c.ServeJSON()
}
```

**Step 2 — Write the Controller**

```go
// controllers/health.go
package controllers

// HealthController handles requests to /api/v1/health.
// It embeds BaseController to inherit RespondJSON and other helpers.
type HealthController struct {
    BaseController
}

// Get handles GET /api/v1/health.
// Beego automatically routes GET requests to a method named Get().
func (c *HealthController) Get() {
    c.RespondJSON(200, true, "Server is running", nil)
}
```

**Step 3 — Define the Router**

```go
// routers/router.go
package routers

import (
    "github.com/beego/beego/v2/server/web"
    "yourmodule/controllers"
)

func init() {
    ns := web.NewNamespace("/api/v1",
        web.NSRouter("/health", &controllers.HealthController{}),
    )
    web.AddNamespace(ns)
}
```

**Step 4 — Start the Server**

```go
// main.go
package main

import (
    _ "yourmodule/routers"
    web "github.com/beego/beego/v2/server/web"
)

func main() {
    web.Run() // reads host and port from conf/app.conf
}
```

## The 5 Key Differences Explained

### 1. Handler: Function → Struct Method

In raw `net/http`, a handler is a plain function:

```go
func healthHandler(w http.ResponseWriter, r *http.Request) { ... }
```

In Beego, a handler is a **method on a struct**:

```go
func (c *HealthController) Get() { ... }
```

The method name corresponds directly to the HTTP verb. Beego calls the right
method automatically based on the incoming request method.

```go
type ProductController struct { BaseController }

func (c *ProductController) Get()    { /* handles GET    */ }
func (c *ProductController) Post()   { /* handles POST   */ }
func (c *ProductController) Put()    { /* handles PUT    */ }
func (c *ProductController) Delete() { /* handles DELETE */ }
```

No `switch r.Method` needed anywhere.

---

### 2. Reading the Request

```go
// Raw net/http
body, _  := io.ReadAll(r.Body)
userID   := r.Header.Get("X-User-ID")
id       := r.PathValue("id")              // or via a router like gorilla/mux
category := r.URL.Query().Get("category")

// Beego — all accessed through c.Ctx.Input
body     := c.Ctx.Input.RequestBody        // []byte request body
userID   := c.Ctx.Input.Header("X-User-ID")
id       := c.Ctx.Input.Param(":id")       // path parameter
category := c.Ctx.Input.Query("category")  // query parameter
```

> **Note:** To use `c.Ctx.Input.RequestBody`, you must set
> `copyrequestbody = true` in `conf/app.conf`. Without it, the body will be empty.

---

### 3. Writing the Response

```go
// Raw net/http — manual every time
w.Header().Set("Content-Type", "application/json")
w.WriteHeader(201)
json.NewEncoder(w).Encode(result)

// Beego — declare what to send, framework handles encoding
c.Ctx.Output.SetStatus(201)
c.Data["json"] = result
c.ServeJSON() // sets Content-Type, encodes to JSON, writes — all at once
```

---

### 4. Routing

```go
// Raw net/http
http.HandleFunc("/api/v1/health",        healthHandler)
http.HandleFunc("/api/v1/users",         usersHandler)   // must switch on r.Method inside
http.HandleFunc("/api/v1/users/{id}",    userByIDHandler)

// Beego — grouped by namespace, method dispatch is automatic
ns := web.NewNamespace("/api/v1",
    web.NSRouter("/health",       &HealthController{}),
    web.NSRouter("/users",        &UserController{}),
    web.NSRouter("/users/:id",    &UserController{}),
)
web.AddNamespace(ns)
```

Namespaces let you group routes under a common prefix cleanly.
You can also nest namespaces (e.g. `/api/v1` inside `/api`).

---

### 5. Configuration and Startup

Raw `net/http` has no built-in configuration system. You manage this yourself.

Beego provides `conf/app.conf`:

```ini
appname  = my-app
httpport = 8080
runmode  = dev
copyrequestbody = true
```

Read any value anywhere in your application:

```go
appName, _ := web.AppConfig.String("appname")
port, _    := web.AppConfig.String("httpport")
```

Start the server — Beego picks up port and mode from the config automatically:

```go
web.Run()
```

## Understanding the Inheritance Chain

This is the structural reason why `BaseController` works the way it does:

```
YourController  (e.g. HealthController)
    └── embeds BaseController            ← your shared helpers live here
            └── embeds web.Controller   ← Beego's built-in
                    └── gives access to:
                            c.Ctx          (request + response context)
                            c.Data         (data to render)
                            c.ServeJSON()  (write JSON response)
                            c.ServeHTML()
                            c.Abort()
                            ... and more
```

`BaseController` sits in the middle by design. You add your own shared methods
there (like `RespondJSON`, auth checks, or logging helpers) so every controller
in your application inherits them without repetition. This follows Go's
composition pattern — there is no class hierarchy, just struct embedding.

---

## A Common Mistake: Forgetting `return` After a Response

In raw `net/http`, writing to `w` does not stop the function. In Beego,
calling `c.ServeJSON()` also does not stop execution. You must `return`
explicitly, or the function continues and may call `ServeJSON()` a second time.

```go
// WRONG — ServeJSON is called twice if the record is not found
func (c *UserController) Get() {
    user, err := models.GetUserByID(id)
    if err != nil {
        c.RespondJSON(404, false, "User not found", nil)
        // execution falls through to the next line!
    }
    c.RespondJSON(200, true, "OK", user)
}

// CORRECT
func (c *UserController) Get() {
    user, err := models.GetUserByID(id)
    if err != nil {
        c.RespondJSON(404, false, "User not found", nil)
        return // stop here
    }
    c.RespondJSON(200, true, "OK", user)
}
```

## Typical Project Structure

```
my-beego-app/
├── conf/
│   └── app.conf          # app name, port, run mode
├── controllers/
│   ├── base.go           # BaseController + shared helpers
│   ├── health.go
│   └── user.go
├── models/
│   └── user.go           # data structs + storage logic
├── routers/
│   └── router.go         # all route definitions
├── main.go
├── go.mod
└── go.sum
```

Controllers handle HTTP concerns only (parsing input, calling models,
returning responses). Models handle data and business logic.
This separation keeps controllers short and testable.

## Quick Reference

| Concept | Raw `net/http` | Beego |
|---|---|---|
| Handler unit | function | struct method |
| HTTP method routing | manual `switch r.Method` | automatic (`Get()`, `Post()`, …) |
| Request body | `io.ReadAll(r.Body)` | `c.Ctx.Input.RequestBody` |
| Request headers | `r.Header.Get(key)` | `c.Ctx.Input.Header(key)` |
| Path parameters | router-dependent | `c.Ctx.Input.Param(":key")` |
| Query parameters | `r.URL.Query().Get(key)` | `c.Ctx.Input.Query(key)` |
| Write JSON response | `json.NewEncoder(w).Encode(v)` | `c.Data["json"] = v; c.ServeJSON()` |
| Set status code | `w.WriteHeader(code)` | `c.Ctx.Output.SetStatus(code)` |
| Configuration | DIY | `conf/app.conf` + `web.AppConfig` |
| Server start | `http.ListenAndServe(addr, nil)` | `web.Run()` |
| Logging | `log.Println` / DIY | `web.Info()`, `web.Warn()`, `web.Error()` |
