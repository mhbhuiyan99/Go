# Installing Beego v2, the Bee Tool, and Initializing a Project

> **Prerequisites:** Go 1.21 or higher installed and `GOPATH`/`GOBIN` configured.
> Verify with `go version` before starting.

---

## Understanding the Two Pieces

Before installing, it helps to know what you are actually installing:

| Tool | What it is | How you use it |
|---|---|---|
| **Beego v2** | The web framework — a Go module your code imports | `go get`, then `import` in code |
| **bee** | A CLI dev tool — hot reload, code generation, building | Terminal commands like `bee run` |

They are separate. Beego is a library. `bee` is a development utility.
You need both.

---

## Step 1 — Install the Bee CLI Tool

```bash
go install github.com/beego/bee/v2@latest
```

This downloads and compiles `bee`, placing the binary in your `GOBIN` directory
(usually `$HOME/go/bin`).

**Verify the install:**

```bash
bee version
```

Expected output (version numbers may differ):

```
______
| ___ \
| |_/ /  ___   ___
| ___ \ / _ \ / _ \
| |_/ /|  __/|  __/
\____/  \___| \___| v2.x.x
```

**If `bee` is not found after install**, your `GOBIN` is not in your `PATH`.
Fix it by adding this to your shell profile (`~/.bashrc`, `~/.zshrc`, etc.):

```bash
export PATH=$PATH:$(go env GOPATH)/bin
```

Then reload: `source ~/.bashrc` (or open a new terminal).

---

## Step 2 — Create Your Project

Navigate to where you keep your Go projects, then run:

```bash
# For a full MVC project (with views, static files):
bee new my-app

# For an API-only project (no views, no static — recommended for REST APIs):
bee new my-app -t api
```

> Use `-t api` for any project that serves only JSON — no HTML templates needed.
> This is the standard choice for REST APIs.

Then enter the project directory:

```bash
cd my-app
```

---

## Step 3 — Install Beego v2 as a Dependency

The `bee new` command creates a `go.mod` file, but does not download Beego
automatically. Run:

```bash
go get github.com/beego/beego/v2@latest
go mod tidy
```

`go mod tidy` removes unused dependencies and ensures `go.sum` is up to date.

**Verify Beego is in your `go.mod`:**

```bash
cat go.mod
```

You should see a line like:

```
require github.com/beego/beego/v2 v2.x.x
```

## Step 4 — Understand the Generated Project Structure

After `bee new my-app -t api`, the structure looks like this:

```
my-app/
├── conf/
│   └── app.conf        # Application configuration (port, run mode, app name)
├── controllers/
│   └── object.go       # Example controller — safe to delete
│   └── user.go         # Example controller — safe to delete
├── models/             # Data structs and storage logic go here
├── routers/
│   └── router.go       # All route definitions go here
├── tests/
│   └── default_test.go # Example test — safe to delete
├── main.go             # Entry point
├── go.mod
└── go.sum
```

### `conf/app.conf` — the configuration file

```ini
appname = my-app
httpport = 8080
runmode = dev
```

| Key | What it does |
|---|---|
| `appname` | Name of the application (used in logs) |
| `httpport` | Port the server listens on |
| `runmode` | `dev` (verbose errors) or `prod` (minimal errors exposed) |

One important setting to add manually for REST APIs:

```ini
copyrequestbody = true
```

Without this, `c.Ctx.Input.RequestBody` will always be empty when you try
to read a POST/PUT request body in a controller.

### `main.go` — the entry point

```go
package main

import (
    _ "my-app/routers"                     // imports routers so init() runs
    beego "github.com/beego/beego/v2/server/web"
)

func main() {
    beego.Run()
}
```

The blank import `_ "my-app/routers"` is important. It triggers the `init()`
function inside `routers/router.go`, which registers all your routes.
Without it, the server starts but has no routes.

### `routers/router.go` — route definitions

```go
package routers

import (
    beego "github.com/beego/beego/v2/server/web"
    "my-app/controllers"
)

func init() {
    beego.Router("/", &controllers.MainController{})
}
```

All routes are registered inside `init()`. Beego calls this automatically
when the package is imported.

## Step 5 — Run the Project

```bash
bee run
```

`bee run` does two things:
1. Starts your Beego server
2. Watches for file changes and **hot-reloads** automatically

You should see output like:

```
______
| ___ \
| |_/ /  ___   ___
| ___ \ / _ \ / _ \
| |_/ /|  __/|  __/
\____/  \___| \___| v2.x.x

2025/01/01 10:00:00 INFO     ▶ 0001 Using 'my-app' as 'appname'
2025/01/01 10:00:00 INFO     ▶ 0002 Initializing watcher...
2025/01/01 10:00:00 SUCCESS  ▶ 0003 Built Successfully!
2025/01/01 10:00:00 INFO     ▶ 0004 Restarting 'my-app'...
2025/01/01 10:00:00 SUCCESS  ▶ 0005 'my-app' is running...
```

Test it:

```bash
curl http://localhost:8080
```

## Step 6 — Manually Initialize a Project (Without `bee new`)

If you prefer to set up the structure yourself — or if `bee new` generates
more than you need — here is the minimal setup:

**1. Initialize the Go module:**

```bash
mkdir my-app && cd my-app
go mod init my-app
```

**2. Create the folder structure:**

```bash
mkdir -p conf controllers models routers
```

**3. Create `conf/app.conf`:**

```ini
appname = my-app
httpport = 8080
runmode = dev
copyrequestbody = true
```

**4. Create `main.go`:**

```go
package main

import (
    _ "my-app/routers"
    web "github.com/beego/beego/v2/server/web"
)

func main() {
    web.Run()
}
```

**5. Create `routers/router.go`:**

```go
package routers

import (
    web "github.com/beego/beego/v2/server/web"
    "my-app/controllers"
)

func init() {
    ns := web.NewNamespace("/api/v1",
        web.NSRouter("/health", &controllers.HealthController{}),
    )
    web.AddNamespace(ns)
}
```

**6. Create `controllers/health.go`:**

```go
package controllers

import web "github.com/beego/beego/v2/server/web"

type HealthController struct {
    web.Controller
}

func (c *HealthController) Get() {
    c.Data["json"] = map[string]interface{}{
        "success": true,
        "message": "Server is running",
    }
    c.ServeJSON()
}
```

**7. Install Beego and run:**

```bash
go get github.com/beego/beego/v2@latest
go mod tidy
bee run
```

Test:

```bash
curl http://localhost:8080/api/v1/health
# {"message":"Server is running","success":true}
```

## Common Problems and Fixes

### `bee: command not found`

Your `GOBIN` is not in `PATH`. Add this to your shell profile and reload:

```bash
export PATH=$PATH:$(go env GOPATH)/bin
```

---

### `go.mod` module name mismatch

If you create the project manually and your `go.mod` says `module my-app`,
all internal imports must use `my-app/...` as the prefix:

```go
import "my-app/controllers"   // correct
import "./controllers"         // wrong — relative imports don't work in Go modules
```

---

### Request body is always empty

Add `copyrequestbody = true` to `conf/app.conf`. This is required for
`c.Ctx.Input.RequestBody` to be populated in POST and PUT handlers.

---

### Port already in use

Change `httpport` in `conf/app.conf` to any free port (e.g. `8081`),
or kill the process using the port:

```bash
# Find and kill the process on port 8080 (Linux/macOS)
lsof -ti:8080 | xargs kill
```

---

### `bee run` rebuilds but changes don't appear

Make sure you saved the file. `bee` watches for file system write events —
some editors write to a temp file and swap, which some watchers miss.
Stopping and restarting `bee run` always works as a fallback.

---

## `bee run` vs `go run main.go`

| | `bee run` | `go run main.go` |
|---|---|---|
| Hot reload on file change | Yes | No |
| Reads `conf/app.conf` | Yes | Yes |
| Suitable for production | No | No |
| For development | Yes | Works, but manual restarts |

Use `bee run` during development. For production, build a binary:

```bash
go build -o my-app .
./my-app
```

## Quick Reference — Commands

```bash
# Install bee CLI
go install github.com/beego/bee/v2@latest

# Create a new API project
bee new my-app -t api

# Install Beego into the project
go get github.com/beego/beego/v2@latest
go mod tidy

# Run with hot reload
bee run

# Build a production binary
go build -o my-app .
```
