# Beego Configuration ŌĆö `conf/app.conf` and `web.AppConfig`

> **Who this is for:** Developers learning Beego who want to understand how
> application configuration works ŌĆö how to define config values, read them
> in code, and manage different environments.

---

## Why a Config File?

Hardcoding values like port numbers, database credentials, or file paths
directly in Go code is a bad practice:

- You have to recompile to change a value
- You can't safely commit credentials to version control
- You can't have different values for development vs production

Beego solves this with a dedicated configuration file: `conf/app.conf`.

```

```

Beego looks for this file **relative to where the binary runs from**.
When you use `bee run`, the working directory is your project root,
so `conf/app.conf` is always found automatically. No path setup needed.

---

## File Format

`app.conf` uses a simple `key = value` format ŌĆö not JSON, not YAML, not TOML.

```ini
# Lines starting with # are comments

appname  = my-app
httpport = 8080
runmode  = dev
```

Rules:
- One key-value pair per line
- `=` separates key and value
- Whitespace around `=` is allowed and ignored
- Keys are **case-insensitive** ŌĆö `httpport` and `HttpPort` are the same
- String values do **not** need quotes
- `#` starts a comment

---

## Beego's Built-in Keys

These keys are recognized and used by Beego itself. You do not need to
write code to apply them ŌĆö Beego reads and acts on them automatically
when the server starts.

| Key | Type | Default | What it does |
|---|---|---|---|
| `appname` | string | `beego` | Application name shown in logs |
| `httpport` | int | `8080` | Port the HTTP server listens on |
| `httpaddr` | string | `""` (all interfaces) | IP address to bind to |
| `runmode` | string | `dev` | `dev` or `prod` ŌĆö affects error verbosity |
| `copyrequestbody` | bool | `false` | Must be `true` to read `c.Ctx.Input.RequestBody` |
| `enabledocs` | bool | `false` | Auto-generate API docs (Swagger) |
| `sessionon` | bool | `false` | Enable session support |
| `maxmemory` | int64 | `1 << 26` | Max memory for multipart form parsing |

> **`copyrequestbody = true` is essential for REST APIs.**
> Without it, reading the request body in a POST or PUT controller will
> always return an empty byte slice.

## Adding Your Own Custom Keys

Beyond Beego's built-in keys, you can store **any application-specific
configuration** in the same file:

```ini
# Beego built-in
appname  = my-app
httpport = 8080
runmode  = dev
copyrequestbody = true

# Your custom keys
datadir      = data
userfile     = users.csv
expensefile  = expenses.csv
maxpagesize  = 50
jwtSecret    = changeme
```

There are no restrictions on key names. Any key Beego does not recognize
is simply ignored by the framework and available for you to read.

## Reading Config Values in Code

Import the Beego web package and use `web.AppConfig`:

```go
import web "github.com/beego/beego/v2/server/web"
```

`web.AppConfig` provides typed getter methods. Each returns `(value, error)`.

### Reading a string

```go
appName, err := web.AppConfig.String("appname")
if err != nil {
    // key not found or unreadable
}
// appName == "my-app"
```

### Reading an int

```go
port, err := web.AppConfig.Int("httpport")
// port == 8080
```

### Reading a bool

```go
copyBody, err := web.AppConfig.Bool("copyrequestbody")
// copyBody == true
```

### Reading a float

```go
rate, err := web.AppConfig.Float("taxrate")
```

### Reading an int64

```go
maxMem, err := web.AppConfig.Int64("maxmemory")
```

## The Full AppConfig API

```go
web.AppConfig.String("key")       // returns (string, error)
web.AppConfig.Int("key")          // returns (int, error)
web.AppConfig.Int64("key")        // returns (int64, error)
web.AppConfig.Bool("key")         // returns (bool, error)
web.AppConfig.Float("key")        // returns (float64, error)
web.AppConfig.Strings("key")      // returns ([]string, error) ŌĆö comma-separated values
web.AppConfig.DefaultString("key", "fallback")  // returns string, never errors
web.AppConfig.DefaultInt("key", 0)              // returns int, never errors
web.AppConfig.DefaultBool("key", false)         // returns bool, never errors
web.AppConfig.DefaultFloat("key", 0.0)          // returns float64, never errors
```

---

## `Default*` Methods ŌĆö Preferred in Most Cases

The `Default*` methods are cleaner for values that always have a sensible
fallback. They never return an error ŌĆö if the key is missing, the fallback
is returned instead.

```go
// Instead of this:
maxSize, err := web.AppConfig.Int("maxpagesize")
if err != nil {
    maxSize = 20  // fallback
}

// Write this:
maxSize := web.AppConfig.DefaultInt("maxpagesize", 20)
```

Use the error-returning methods when a missing key should be treated as
a fatal configuration error (e.g. a required secret that has no default).

## Practical Pattern ŌĆö Load Config Once at Startup

A common pattern is to load all config values once when the application
starts and store them in a package-level struct. This avoids repeated
config lookups throughout the codebase.

```go
// config/config.go
package config

import (
    "log"
    web "github.com/beego/beego/v2/server/web"
)

// AppConfig holds all application configuration values.
type AppConfig struct {
    DataDir     string
    UserFile    string
    MaxPageSize int
}

// Config is the global config instance, loaded once at startup.
var Config AppConfig

// Load reads all values from conf/app.conf.
// Call this once from main() before starting the server.
func Load() {
    Config = AppConfig{
        DataDir:     web.AppConfig.DefaultString("datadir", "data"),
        UserFile:    web.AppConfig.DefaultString("userfile", "users.csv"),
        MaxPageSize: web.AppConfig.DefaultInt("maxpagesize", 20),
    }

    // For required values with no safe default, fail fast
    secret, err := web.AppConfig.String("jwtsecret")
    if err != nil || secret == "" {
        log.Fatal("FATAL: jwtsecret is not set in conf/app.conf")
    }
}
```

```go
// main.go
package main

import (
    _ "my-app/routers"
    "my-app/config"
    web "github.com/beego/beego/v2/server/web"
)

func main() {
    config.Load()
    web.Run()
}
```

```go
// anywhere in your app
import "my-app/config"

filePath := config.Config.DataDir + "/" + config.Config.UserFile
// "data/users.csv"
```

## Multiple Environments ŌĆö `[section]` Blocks

`app.conf` supports named sections to define different values for
different environments. This is Beego's built-in approach to
dev/staging/production config.

```ini
appname  = my-app
httpport = 8080
runmode  = dev

[dev]
datadir  = data_dev
loglevel = debug

[prod]
datadir  = data
loglevel = warn
httpport = 80
```

Beego automatically activates the section that matches `runmode`.
When `runmode = dev`, values in `[dev]` override the top-level values.
When `runmode = prod`, values in `[prod]` take effect.

Read section-specific values the same way ŌĆö Beego applies the active
section automatically:

```go
// runmode = dev ŌåÆ reads from [dev] block, falls back to top-level
dataDir, _ := web.AppConfig.String("datadir")
// returns "data_dev" in dev, "data" in prod
```

Override `runmode` at runtime without changing the file:

```bash
# Start in production mode regardless of what app.conf says
bee run -runmode=prod
```

## Setting Config Values Programmatically

You can set or override config values in code. This is useful in tests
or when you need to derive a value from other sources:

```go
web.AppConfig.Set("datadir", "/tmp/test-data")

// Read it back normally
dir, _ := web.AppConfig.String("datadir")
// "/tmp/test-data"
```

This does not write to the file ŌĆö it only affects the in-memory config
for the current process.

## Using Config in Tests

In unit tests, Beego may not automatically find `conf/app.conf` because
the working directory is the package directory, not the project root.

**Option 1 ŌĆö Set values programmatically in `TestMain`:**

```go
func TestMain(m *testing.M) {
    web.AppConfig.Set("datadir", "testdata")
    web.AppConfig.Set("userfile", "users_test.csv")
    os.Exit(m.Run())
}
```

**Option 2 ŌĆö Load a test-specific config file:**

```go
import "github.com/beego/beego/v2/core/config"

func TestMain(m *testing.M) {
    cfg, _ := config.NewConfig("ini", "../../conf/app.conf")
    web.BConfig.AppName = cfg.DefaultString("appname", "test")
    os.Exit(m.Run())
}
```

Option 1 is simpler for most test suites.

## What Happens if `app.conf` Is Missing?

Beego does not crash. It falls back to built-in defaults:

| Key | Default fallback |
|---|---|
| `httpport` | `8080` |
| `runmode` | `dev` |
| `appname` | `beego` |

Your own custom keys will return errors when read ŌĆö which is why
`Default*` methods or a fail-fast startup check are important.

## Common Mistakes

### Reading config before Beego has loaded it

`web.AppConfig` is populated when the Beego package is initialized.
This happens before `main()` runs, so config is always available by
the time you call `web.Run()`. No special initialization needed.

```go
// This is safe ŌĆö AppConfig is already populated here
func main() {
    port, _ := web.AppConfig.String("httpport") // works fine
    web.Run()
}
```

### Quoting string values

String values in `app.conf` do **not** use quotes.

```ini
# WRONG ŌĆö the quotes become part of the string value
datadir = "data"

# CORRECT
datadir = data
```

If you read `"data"` back, you will get the literal string `"data"` with
the quote characters included.

### Case sensitivity in key names

Keys are case-insensitive when read, but it's best to be consistent.
Pick one style (all lowercase is conventional) and stick to it:

```go
// Both of these read the same key
web.AppConfig.String("httpport")
web.AppConfig.String("HttpPort")
```
