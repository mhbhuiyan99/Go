# Beego v2 Routing — `router.go`, Namespaces, and Method Mapping

> **Who this is for:** Go developers learning Beego who want to understand
> how URL routes are defined, how HTTP methods map to controller functions,
> and how to organize routes cleanly using namespaces.

---

## How Beego Routing Works — The Big Picture

In raw `net/http`, there is **no enforced convention** for where routes are
registered. `http.HandleFunc()` and `mux.Handle()` are just function calls —
you can make them from `main()`, a dedicated `routes.go`, a `bootstrap`
package, a domain module, or anywhere your architecture decides. DDD, Clean
Architecture, flat scripts — all are valid.

Beego, by contrast, establishes a **specific convention**: routes belong in
`routers/router.go` inside an `init()` function, and the framework is built
around that expectation.

The real difference is not *where* — it is **freedom vs convention**:

| | raw `net/http` | Beego |
|---|---|---|
| Where routes go | Anywhere you decide | `routers/router.go` by convention |
| Structure enforced? | No — you design it | Yes — framework expects it |
| `init()` pattern required? | No | Yes — because of the blank import chain |

Go calls `init()` automatically before `main()` runs — as long as the package is imported.

```
main.go
  └── imports _ "my-app/routers"    ← blank import triggers init()
                    └── init() runs
                            └── registers all routes with Beego
                                    └── web.Run() starts the server
```

This is why `main.go` always has a blank import of the routers package:

```go
// main.go
package main

import (
    _ "my-app/routers"   // must be here — triggers route registration
    web "github.com/beego/beego/v2/server/web"
)

func main() {
    web.Run()
}
```

Remove the blank import and your server starts but has zero routes registered.

## Two Ways to Define Routes

Beego provides two routing styles. You will almost always use **namespaces**.

| Style | Function | Best for |
|---|---|---|
| Simple router | `web.Router()` | Quick single routes, prototyping |
| Namespace router | `web.NewNamespace()` + `web.NSRouter()` | Any real application |

---

## Style 1 — Simple Router (`web.Router`)

```go
// routers/router.go
package routers

import (
    web "github.com/beego/beego/v2/server/web"
    "my-app/controllers"
)

func init() {
    web.Router("/users",     &controllers.UserController{})
    web.Router("/users/:id", &controllers.UserController{})
}
```

This works but has a problem at scale — all routes are flat, there's
no grouping, and adding a prefix like `/api/v1` to everything means
repeating it on every line.

---

## Style 2 — Namespace Router (Recommended)

Namespaces let you group routes under a common URL prefix and
nest groups inside each other.

```go
func init() {
    ns := web.NewNamespace("/api/v1",
        web.NSRouter("/health",   &controllers.HealthController{}),
        web.NSRouter("/users",    &controllers.UserController{}),
        web.NSRouter("/users/:id",&controllers.UserController{}),
    )
    web.AddNamespace(ns)
}
```

`web.AddNamespace(ns)` is required — it registers the namespace with
Beego's router. Forgetting it means none of the routes inside exist.

## How HTTP Methods Map to Controller Methods

This is the core of Beego routing. When a request arrives, Beego looks
at the **HTTP method** and calls the matching **Go method** on the controller.

The mapping is by **convention** — method names must match exactly:

| HTTP Method | Controller Method |
|---|---|
| `GET` | `Get()` |
| `POST` | `Post()` |
| `PUT` | `Put()` |
| `DELETE` | `Delete()` |
| `PATCH` | `Patch()` |
| `HEAD` | `Head()` |
| `OPTIONS` | `Options()` |

Example — a `UserController` that handles all CRUD operations:

```go
// controllers/user.go
package controllers

type UserController struct {
    BaseController
}

// Handles GET /api/v1/users  and  GET /api/v1/users/:id
func (c *UserController) Get() {
    // if :id is in the URL, get one user
    // otherwise, list all users
}

// Handles POST /api/v1/users
func (c *UserController) Post() {
    // create a new user
}

// Handles PUT /api/v1/users/:id
func (c *UserController) Put() {
    // update a user
}

// Handles DELETE /api/v1/users/:id
func (c *UserController) Delete() {
    // delete a user
}
```

You do not need to implement every method. If a `DELETE` request arrives
for a route whose controller has no `Delete()` method, Beego returns
`405 Method Not Allowed` automatically.

---

## Explicit Method Mapping (Override the Convention)

Sometimes you need a route that does not follow the convention — for example,
a `GET` request that should call a method named `Summary()` instead of `Get()`.

Use `web.NSRouter` with an explicit method map as the third argument:

```go
web.NSRouter("/users/summary", &controllers.UserController{}, "get:Summary")
//                                                             ↑
//                           "httpmethod:ControllerMethodName"
```

The format is `"httpmethod:MethodName"`. You can map multiple methods
on one route by separating with semicolons:

```go
web.NSRouter("/users/export",
    &controllers.UserController{},
    "get:Export;post:ImportUsers",
)
```

This is especially important when **two routes share the same controller**
but need different methods called:

```go
// Without explicit mapping — both GET requests call Get()
web.NSRouter("/expenses",      &controllers.ExpenseController{})
web.NSRouter("/expenses/:id",  &controllers.ExpenseController{})

// With explicit mapping — named methods, much clearer intent
web.NSRouter("/expenses",          &controllers.ExpenseController{}, "get:List;post:Create")
web.NSRouter("/expenses/:id",      &controllers.ExpenseController{}, "get:GetOne;put:Update;delete:Remove")
web.NSRouter("/expenses/summary",  &controllers.ExpenseController{}, "get:Summary")
```

```go
// controllers/expense.go
func (c *ExpenseController) List()    { /* GET /expenses        */ }
func (c *ExpenseController) Create()  { /* POST /expenses       */ }
func (c *ExpenseController) GetOne()  { /* GET /expenses/:id    */ }
func (c *ExpenseController) Update()  { /* PUT /expenses/:id    */ }
func (c *ExpenseController) Remove()  { /* DELETE /expenses/:id */ }
func (c *ExpenseController) Summary() { /* GET /expenses/summary*/ }
```

This approach is **more readable and maintainable** than relying solely
on convention, especially when a controller has many endpoints.

## Path Parameters

Define a path parameter with `:name` syntax in the route:

```go
web.NSRouter("/users/:id", &controllers.UserController{})
```

Read it inside the controller with `c.Ctx.Input.Param(":id")`:

```go
func (c *UserController) Get() {
    id := c.Ctx.Input.Param(":id")
    // id is a string — convert with strconv.Atoi() if you need an int
}
```

### Multiple path parameters

```go
web.NSRouter("/teams/:teamID/members/:memberID", &controllers.MemberController{})
```

```go
teamID   := c.Ctx.Input.Param(":teamID")
memberID := c.Ctx.Input.Param(":memberID")
```

### Wildcard path parameter

`:splat` or `*` captures everything after a prefix:

```go
web.NSRouter("/files/*", &controllers.FileController{})
// GET /files/docs/report.pdf  →  c.Ctx.Input.Param(":splat") == "docs/report.pdf"
```

## Route Order and Conflicts — Static vs Dynamic

**Static routes always take priority over dynamic routes**, regardless of
the order they are defined:

```go
// Even if defined second, /expenses/summary matches before /expenses/:id
web.NSRouter("/expenses/summary", &controllers.ExpenseController{}, "get:Summary")
web.NSRouter("/expenses/:id",     &controllers.ExpenseController{}, "get:GetOne")
```

`GET /expenses/summary` → calls `Summary()` ✓
`GET /expenses/123`     → calls `GetOne()` ✓

If you relied on order and put `/expenses/:id` first, Beego's static-first
priority still routes `/expenses/summary` to `Summary()` correctly.
But explicitly defining static routes before dynamic ones is good style.

## Nested Namespaces

Namespaces can be nested for deeper groupings:

```go
func init() {
    v1 := web.NewNamespace("/api/v1",
        // Auth routes
        web.NSNamespace("/auth",
            web.NSRouter("/register", &controllers.AuthController{}, "post:Register"),
            web.NSRouter("/login",    &controllers.AuthController{}, "post:Login"),
        ),
        // User routes
        web.NSNamespace("/users",
            web.NSRouter("",      &controllers.UserController{}, "get:List;post:Create"),
            web.NSRouter("/:id",  &controllers.UserController{}, "get:GetOne;put:Update;delete:Remove"),
        ),
        // Health check
        web.NSRouter("/health", &controllers.HealthController{}),
    )
    web.AddNamespace(v1)
}
```

This produces:

```
POST   /api/v1/auth/register
POST   /api/v1/auth/login
GET    /api/v1/users
POST   /api/v1/users
GET    /api/v1/users/:id
PUT    /api/v1/users/:id
DELETE /api/v1/users/:id
GET    /api/v1/health
```

Clean, no repetition, and the structure mirrors the URL hierarchy.

## Namespace Middleware (Filters)

You can attach a middleware function to a namespace so it runs before
every route inside it. This is useful for auth checks:

```go
func init() {
    // Public routes — no auth needed
    public := web.NewNamespace("/api/v1",
        web.NSRouter("/health",         &controllers.HealthController{}),
        web.NSRouter("/auth/register",  &controllers.AuthController{}, "post:Register"),
        web.NSRouter("/auth/login",     &controllers.AuthController{}, "post:Login"),
    )

    // Protected routes — run AuthFilter before every handler
    protected := web.NewNamespace("/api/v1",
        web.NSBefore(AuthFilter),
        web.NSRouter("/users",    &controllers.UserController{}, "get:List"),
        web.NSRouter("/users/:id",&controllers.UserController{}, "get:GetOne"),
    )

    web.AddNamespace(public, protected)
}

// AuthFilter runs before every route in the protected namespace.
// It checks for an auth header and stops the request if invalid.
func AuthFilter(ctx *context.Context) {
    userID := ctx.Input.Header("X-User-ID")
    if userID == "" {
        ctx.Output.SetStatus(401)
        ctx.Output.JSON(map[string]interface{}{
            "success": false,
            "message": "Unauthorized",
        }, false, false)
    }
}
```

`web.NSBefore(fn)` runs the function before the route handler.
`web.NSAfter(fn)` runs it after.

## Printing All Registered Routes

During development, it's useful to verify that all your routes are
registered as expected. Add this to `main.go` before `web.Run()`:

```go
// main.go
func main() {
    config.Load()

    // Print all registered routes to the console
    web.BeeApp.Handlers.PrintTree()

    web.Run()
}
```

Or run the server and hit the built-in admin port (enabled in `app.conf`):

```ini
enableadmin = true
adminport   = 8088
```

Then visit `http://localhost:8088/qps` in a browser to see live route info.

## Putting It All Together — A Complete `router.go`

```go
// routers/router.go
package routers

import (
    web "github.com/beego/beego/v2/server/web"
    "my-app/controllers"
)

func init() {
    ns := web.NewNamespace("/api/v1",

        // Health
        web.NSRouter("/health",
            &controllers.HealthController{}),

        // Auth (public)
        web.NSNamespace("/auth",
            web.NSRouter("/register",
                &controllers.AuthController{}, "post:Register"),
            web.NSRouter("/login",
                &controllers.AuthController{}, "post:Login"),
        ),

        // Users (protected)
        web.NSNamespace("/users",
            web.NSRouter("",
                &controllers.UserController{}, "get:List;post:Create"),
            web.NSRouter("/:id",
                &controllers.UserController{}, "get:GetOne;put:Update;delete:Remove"),
        ),
    )

    web.AddNamespace(ns)
}
```

## Common Mistakes

### Forgetting `web.AddNamespace(ns)`

```go
func init() {
    ns := web.NewNamespace("/api/v1",
        web.NSRouter("/health", &controllers.HealthController{}),
    )
    // web.AddNamespace(ns) ← forgot this
    // Result: server starts, all routes return 404
}
```

### Forgetting the blank import in `main.go`

```go
// main.go
import (
    // _ "my-app/routers"  ← missing
    web "github.com/beego/beego/v2/server/web"
)
// Result: server starts, zero routes registered, all 404
```

### Conflicting static and dynamic routes on the same path

```go
// PROBLEM — two routes that match the same URL
web.NSRouter("/expenses/summary", &controllers.ExpenseController{}, "get:Summary")
web.NSRouter("/expenses/:id",     &controllers.ExpenseController{}, "get:GetOne")
// GET /expenses/summary — which one wins?
// Answer: the static route (/expenses/summary) always wins in Beego.
// This is actually fine — just make sure you know the rule.
```

### Using the wrong HTTP method name in explicit mapping

```go
// WRONG — method names must be lowercase in the mapping string
web.NSRouter("/users", &controllers.UserController{}, "GET:List")

// CORRECT
web.NSRouter("/users", &controllers.UserController{}, "get:List")
```

## Quick Reference

```go
// routers/router.go
package routers

import (
    web "github.com/beego/beego/v2/server/web"
    "my-app/controllers"
)

func init() {
    ns := web.NewNamespace("/api/v1",

        // Convention-based (GET→Get, POST→Post, …)
        web.NSRouter("/items", &controllers.ItemController{}),

        // Explicit method mapping
        web.NSRouter("/items/:id",
            &controllers.ItemController{},
            "get:GetOne;put:Update;delete:Remove"),

        // Nested namespace
        web.NSNamespace("/auth",
            web.NSRouter("/login",    &controllers.AuthController{}, "post:Login"),
            web.NSRouter("/register", &controllers.AuthController{}, "post:Register"),
        ),
    )
    web.AddNamespace(ns)
}
```

```go
// Reading a path parameter in a controller
id := c.Ctx.Input.Param(":id")
```

| Concept | Code |
|---|---|
| Create namespace | `web.NewNamespace("/prefix", ...)` |
| Add route to namespace | `web.NSRouter("/path", &Controller{})` |
| Explicit method map | `"get:MethodName;post:OtherMethod"` |
| Nested namespace | `web.NSNamespace("/sub", ...)` |
| Namespace middleware | `web.NSBefore(filterFunc)` |
| Register namespace | `web.AddNamespace(ns)` |
| Path parameter in route | `/items/:id` |
| Read path parameter | `c.Ctx.Input.Param(":id")` |
