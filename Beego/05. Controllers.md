# Beego Controllers — Embedding, Handler Methods, and JSON Responses

> **Who this is for:** Go developers learning Beego who want to understand
> how controllers are structured, how embedding works, how to write handler
> methods, and how to return consistent JSON responses.

---

## What Is a Controller in Beego?

In raw `net/http`, a handler can be a plain function **or** a struct method —
`net/http` supports both. If you implement the `http.Handler` interface on a
struct, you can use methods just fine:

```go
// Plain function approach
func getUser(w http.ResponseWriter, r *http.Request) { ... }

// Struct method approach — also valid in net/http
type UserHandler struct{}

func (h *UserHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) { ... }
```

The real difference between `net/http` and Beego is not about functions vs
structs. It is about **HTTP verb dispatch**:

In `net/http`, regardless of which approach you choose, you are responsible
for checking the HTTP method yourself and routing to the right logic:

```go
func (h *UserHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    switch r.Method {
    case http.MethodGet:
        h.list(w, r)
    case http.MethodPost:
        h.create(w, r)
    default:
        http.Error(w, "method not allowed", 405)
    }
}
```

In Beego, you never write that switch. You define a struct that embeds
`web.Controller`, name your methods after HTTP verbs, and Beego dispatches
automatically:

```go
type UserController struct {
    web.Controller
}

func (c *UserController) Get()  { ... } // called automatically on GET
func (c *UserController) Post() { ... } // called automatically on POST
```

The struct is called a **controller**. Each controller typically represents
one resource (users, products, orders). Its methods represent the operations
on that resource.

## `web.Controller` — The Foundation

Every Beego controller must embed `web.Controller` from the Beego package:

```go
import web "github.com/beego/beego/v2/server/web"

type UserController struct {
    web.Controller   // ← embed this
}
```

Embedding gives your controller access to everything Beego provides:

| Field / Method | What it gives you |
|---|---|
| `c.Ctx` | The full request + response context |
| `c.Ctx.Input` | Read request data (body, headers, params, query) |
| `c.Ctx.Output` | Write response data (status code, headers) |
| `c.Data` | Map to hold data before rendering |
| `c.ServeJSON()` | Encode `c.Data["json"]` and write JSON response |
| `c.Abort("404")` | Stop execution and return an HTTP error |
| `c.GetString("key")` | Read a form or query value |

You never instantiate `web.Controller` manually. Beego does it for each
incoming request. Your job is just to embed it and write methods.

## The `BaseController` Pattern

In any real application, every controller shares common behaviour —
returning a consistent JSON response shape, checking auth headers,
or logging. Rather than repeating this in every controller, define a
`BaseController` that embeds `web.Controller` and holds your shared helpers.
All your other controllers then embed `BaseController` instead of
`web.Controller` directly.

```
web.Controller          ← Beego's built-in
    └── BaseController  ← your shared helpers live here
            └── UserController
            └── ProductController
            └── AuthController
            └── (every other controller)
```

This is **standard Beego practice**, not specific to any project type.

```go
// controllers/base.go
package controllers

import web "github.com/beego/beego/v2/server/web"

// BaseController is embedded by every controller in the application.
// It provides shared response helpers and common behaviour.
type BaseController struct {
    web.Controller
}

// response is the standard JSON envelope for all API responses.
type response struct {
    Success bool        `json:"success"`
    Message string      `json:"message"`
    Data    interface{} `json:"data,omitempty"`
}

// RespondJSON writes a JSON response with the given HTTP status code.
// All controllers should use this instead of writing responses manually.
func (c *BaseController) RespondJSON(status int, success bool, message string, data interface{}) {
    c.Ctx.Output.SetStatus(status)
    c.Data["json"] = response{
        Success: success,
        Message: message,
        Data:    data,
    }
    c.ServeJSON()
}

// RespondError is a convenience wrapper for error responses.
func (c *BaseController) RespondError(status int, message string) {
    c.RespondJSON(status, false, message, nil)
}

// RespondOK is a convenience wrapper for success responses.
func (c *BaseController) RespondOK(message string, data interface{}) {
    c.RespondJSON(200, true, message, data)
}
```

## Writing a Controller

With `BaseController` defined, every other controller is clean and focused
only on its own logic:

```go
// controllers/user.go
package controllers

import (
    "encoding/json"
    "my-app/models"
)

// UserController handles all requests to /api/v1/users
type UserController struct {
    BaseController   // ← embed BaseController, not web.Controller directly
}

// Get handles GET /api/v1/users
// Returns a list of all users.
func (c *UserController) Get() {
    users, err := models.GetAllUsers()
    if err != nil {
        c.RespondError(500, "Failed to retrieve users")
        return
    }
    c.RespondOK("Users retrieved", users)
}

// Post handles POST /api/v1/users
// Creates a new user from the request body.
func (c *UserController) Post() {
    var input struct {
        Name  string `json:"name"`
        Email string `json:"email"`
    }

    if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
        c.RespondError(400, "Invalid request body")
        return
    }

    user, err := models.CreateUser(input.Name, input.Email)
    if err != nil {
        c.RespondError(500, "Failed to create user")
        return
    }

    c.RespondJSON(201, true, "User created", user)
}
```

## Method Names — Convention vs Explicit

### Convention-based (HTTP verb → method name)

Beego automatically calls the method whose name matches the HTTP verb:

```go
func (c *UserController) Get()    { } // called on GET
func (c *UserController) Post()   { } // called on POST
func (c *UserController) Put()    { } // called on PUT
func (c *UserController) Delete() { } // called on DELETE
func (c *UserController) Patch()  { } // called on PATCH
```

### Explicit mapping (custom method names)

When a controller handles multiple distinct operations on the same resource,
custom method names are clearer. Register them with explicit mapping in
`router.go`:

```go
// router.go
web.NSRouter("/users",         &controllers.UserController{}, "get:List;post:Create")
web.NSRouter("/users/:id",     &controllers.UserController{}, "get:GetOne;put:Update;delete:Remove")
web.NSRouter("/users/summary", &controllers.UserController{}, "get:Summary")
```

```go
// controllers/user.go
func (c *UserController) List()    { } // GET    /users
func (c *UserController) Create()  { } // POST   /users
func (c *UserController) GetOne()  { } // GET    /users/:id
func (c *UserController) Update()  { } // PUT    /users/:id
func (c *UserController) Remove()  { } // DELETE /users/:id
func (c *UserController) Summary() { } // GET    /users/summary
```

This is more readable than six methods named after HTTP verbs, especially
when a controller grows large.

## Reading Request Data Inside a Controller

All request data is accessed through `c.Ctx.Input`:

### Request body (POST / PUT)

```go
// c.Ctx.Input.RequestBody is a []byte
// Requires copyrequestbody = true in conf/app.conf

var input struct {
    Name  string  `json:"name"`
    Price float64 `json:"price"`
}

if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
    c.RespondError(400, "Invalid JSON")
    return
}
```

### Path parameters

```go
// Route: /users/:id
id := c.Ctx.Input.Param(":id")       // always a string
// Convert to int if needed:
userID, err := strconv.Atoi(id)
if err != nil {
    c.RespondError(400, "Invalid ID")
    return
}
```

### Query parameters

```go
// GET /users?role=admin&page=2
role := c.Ctx.Input.Query("role")    // "admin"
page := c.Ctx.Input.Query("page")    // "2" — always a string
```

### Request headers

```go
token  := c.Ctx.Input.Header("Authorization")
userID := c.Ctx.Input.Header("X-User-ID")
```

## Returning JSON Responses

Beego's JSON response mechanism has three parts:

```go
c.Ctx.Output.SetStatus(201)      // 1. set the HTTP status code
c.Data["json"] = someValue       // 2. put the value to encode
c.ServeJSON()                    // 3. encode as JSON and write the response
```

`c.Data` is a `map[string]interface{}`. The key `"json"` is what
`c.ServeJSON()` looks for. You can put any value there — a struct, a map,
a slice, or a primitive.

```go
// Struct
c.Data["json"] = User{ID: 1, Name: "Alice"}

// Map
c.Data["json"] = map[string]interface{}{"success": true, "count": 42}

// Slice
c.Data["json"] = []User{user1, user2}

// Primitive
c.Data["json"] = "hello"
```

`c.ServeJSON()` sets `Content-Type: application/json` automatically.
You do not need to set it manually.

## HTTP Status Codes — When to Use What

```go
c.Ctx.Output.SetStatus(200)  // OK — successful GET, PUT, DELETE
c.Ctx.Output.SetStatus(201)  // Created — successful POST that created a resource
c.Ctx.Output.SetStatus(400)  // Bad Request — malformed input, validation failure
c.Ctx.Output.SetStatus(401)  // Unauthorized — missing or invalid credentials
c.Ctx.Output.SetStatus(403)  // Forbidden — authenticated but not allowed
c.Ctx.Output.SetStatus(404)  // Not Found — resource does not exist
c.Ctx.Output.SetStatus(409)  // Conflict — e.g. duplicate email on registration
c.Ctx.Output.SetStatus(500)  // Internal Server Error — unexpected failure
```

If you do not call `SetStatus`, Beego defaults to `200`.

---

## The `return` Rule — Critical

`c.ServeJSON()` writes the response but does **not** stop the function.
You must `return` immediately after every response in a conditional branch,
or execution continues and `ServeJSON()` is called a second time.

```go
// WRONG — ServeJSON is called twice when err != nil
func (c *UserController) GetOne() {
    user, err := models.GetUserByID(id)
    if err != nil {
        c.RespondError(404, "User not found")
        // no return — falls through to the next line
    }
    c.RespondOK("User retrieved", user)  // called even on error
}

// CORRECT
func (c *UserController) GetOne() {
    user, err := models.GetUserByID(id)
    if err != nil {
        c.RespondError(404, "User not found")
        return   // stop here
    }
    c.RespondOK("User retrieved", user)
}
```

A good mental rule: **every call to `RespondJSON`, `RespondError`, or
`RespondOK` must be followed by `return`**, unless it is the very last
statement in the function.

---

## Using `c.Abort()` for Immediate Stops

`c.Abort("statusCode")` stops execution immediately and returns an HTTP
error — no need for `return` after it:

```go
func (c *UserController) GetOne() {
    user, err := models.GetUserByID(id)
    if err != nil {
        c.Abort("404")   // stops immediately, no return needed
    }
    c.RespondOK("User retrieved", user)
}
```

However, `c.Abort()` only accepts string status codes and returns Beego's
default error page body — not your custom JSON format. For REST APIs,
`RespondError` + `return` gives you full control over the response body.
Use `c.Abort()` sparingly.

## A Complete Controller Example

```go
// controllers/product.go
package controllers

import (
    "encoding/json"
    "strconv"
    "my-app/models"
)

// ProductController handles CRUD operations for products.
type ProductController struct {
    BaseController
}

// List handles GET /api/v1/products
func (c *ProductController) List() {
    category := c.Ctx.Input.Query("category")

    products, err := models.GetProducts(category)
    if err != nil {
        c.RespondError(500, "Failed to retrieve products")
        return
    }
    c.RespondOK("Products retrieved", products)
}

// Create handles POST /api/v1/products
func (c *ProductController) Create() {
    var input struct {
        Name     string  `json:"name"`
        Price    float64 `json:"price"`
        Category string  `json:"category"`
    }

    if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
        c.RespondError(400, "Invalid request body")
        return
    }

    if input.Name == "" {
        c.RespondError(400, "Name is required")
        return
    }
    if input.Price <= 0 {
        c.RespondError(400, "Price must be greater than zero")
        return
    }

    product, err := models.CreateProduct(input.Name, input.Price, input.Category)
    if err != nil {
        c.RespondError(500, "Failed to create product")
        return
    }

    c.RespondJSON(201, true, "Product created", product)
}

// GetOne handles GET /api/v1/products/:id
func (c *ProductController) GetOne() {
    id, err := strconv.Atoi(c.Ctx.Input.Param(":id"))
    if err != nil {
        c.RespondError(400, "Invalid product ID")
        return
    }

    product, err := models.GetProductByID(id)
    if err != nil {
        c.RespondError(404, "Product not found")
        return
    }

    c.RespondOK("Product retrieved", product)
}

// Update handles PUT /api/v1/products/:id
func (c *ProductController) Update() {
    id, err := strconv.Atoi(c.Ctx.Input.Param(":id"))
    if err != nil {
        c.RespondError(400, "Invalid product ID")
        return
    }

    var input struct {
        Name  string  `json:"name"`
        Price float64 `json:"price"`
    }

    if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
        c.RespondError(400, "Invalid request body")
        return
    }

    product, err := models.UpdateProduct(id, input.Name, input.Price)
    if err != nil {
        c.RespondError(404, "Product not found")
        return
    }

    c.RespondOK("Product updated", product)
}

// Remove handles DELETE /api/v1/products/:id
func (c *ProductController) Remove() {
    id, err := strconv.Atoi(c.Ctx.Input.Param(":id"))
    if err != nil {
        c.RespondError(400, "Invalid product ID")
        return
    }

    if err := models.DeleteProduct(id); err != nil {
        c.RespondError(404, "Product not found")
        return
    }

    c.RespondOK("Product deleted", nil)
}
```

## Controller Lifecycle — What Beego Does Behind the Scenes

When a request arrives, Beego does this automatically:

```
1. Match the URL to a registered route
2. Identify the HTTP method (GET, POST, ...)
3. Create a new instance of the controller struct
4. Populate c.Ctx with the request and response objects
5. Call the matching handler method (Get, Post, or explicitly mapped name)
6. After the method returns, finalize the response
```

A **new controller instance is created for every request**. There is no
shared state between requests inside the controller struct itself. If you
need shared state (e.g. a counter, a cache), it belongs in the model layer
or a package-level variable — not in the controller struct.

## Common Mistakes

### Not embedding `web.Controller`

```go
// WRONG — missing the embedding
type UserController struct{}

// CORRECT
type UserController struct {
    web.Controller
}
// or, with BaseController:
type UserController struct {
    BaseController
}
```

Without the embedding, `c.Ctx`, `c.Data`, `c.ServeJSON()` — nothing exists.
The code will not compile.

### Forgetting `copyrequestbody = true`

```go
// This will always be an empty []byte without copyrequestbody = true in app.conf
json.Unmarshal(c.Ctx.Input.RequestBody, &input)
```

Add to `conf/app.conf`:
```ini
copyrequestbody = true
```

### Calling `SetStatus` after `ServeJSON`

```go
// WRONG — status must be set BEFORE ServeJSON
c.Data["json"] = result
c.ServeJSON()
c.Ctx.Output.SetStatus(201)  // too late, headers already written

// CORRECT
c.Ctx.Output.SetStatus(201)
c.Data["json"] = result
c.ServeJSON()
```

Always set the status code first.

## Quick Reference

```go
// Embed
type MyController struct { BaseController }

// Handler methods (convention-based)
func (c *MyController) Get()    { }
func (c *MyController) Post()   { }
func (c *MyController) Put()    { }
func (c *MyController) Delete() { }

// Read request
body    := c.Ctx.Input.RequestBody          // []byte
id      := c.Ctx.Input.Param(":id")         // path param — string
search  := c.Ctx.Input.Query("q")           // query param — string
token   := c.Ctx.Input.Header("Authorization") // header — string

// Write response
c.Ctx.Output.SetStatus(201)  // set status FIRST
c.Data["json"] = anyValue    // set data SECOND
c.ServeJSON()                // write THIRD — always last
return                       // return AFTER (unless last statement)

// BaseController helpers (if defined)
c.RespondJSON(201, true, "Created", data)
c.RespondError(400, "Bad request")
c.RespondOK("Success", data)
```
