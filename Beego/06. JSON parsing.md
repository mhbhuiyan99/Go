# Parsing JSON Request Bodies in Beego

> **Who this is for:** Go developers learning Beego who want to understand
> how to read and parse a JSON request body sent by a client in POST, PUT,
> and PATCH requests.

---

## The One Required Config Setting

Before anything else — open `conf/app.conf` and confirm this line exists:

```ini
copyrequestbody = true
```

By default, Beego reads the request body once internally and discards it.
`copyrequestbody = true` tells Beego to keep a copy in memory so your
controller can access it via `c.Ctx.Input.RequestBody`.

Without this setting, `c.Ctx.Input.RequestBody` is always an empty `[]byte`,
and every `json.Unmarshal` call silently produces a zero-value struct with
no error — one of the harder bugs to spot.

---

## What `c.Ctx.Input.RequestBody` Is

```go
c.Ctx.Input.RequestBody  // type: []byte
```

It holds the raw bytes of the HTTP request body exactly as the client sent
them. No parsing has happened yet. It is your job to decode those bytes into
a Go value using `json.Unmarshal`.

```
Client sends:                       You receive:
{"name":"Alice","age":30}  →→→  []byte(`{"name":"Alice","age":30}`)
```

---

## The Basic Pattern

```go
import "encoding/json"

func (c *UserController) Create() {
    // 1. Define a struct that matches the expected JSON shape
    var input struct {
        Name  string `json:"name"`
        Email string `json:"email"`
    }

    // 2. Unmarshal the raw bytes into the struct
    if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
        c.RespondError(400, "Invalid JSON")
        return
    }

    // 3. Use the parsed values
    web.Info("Creating user:", input.Name, input.Email)
}
```

Three steps, always in this order:
1. Define the shape you expect
2. Unmarshal into it
3. Validate and use the values

## Defining Input Structs

### Inline (anonymous struct)

For simple, one-off use inside a single controller method:

```go
var input struct {
    Title  string  `json:"title"`
    Amount float64 `json:"amount"`
}
json.Unmarshal(c.Ctx.Input.RequestBody, &input)
```

### Named struct (separate type)

When the same shape is used in multiple places, or when the struct is complex:

```go
// models/product.go  or  controllers/product.go
type CreateProductInput struct {
    Name     string  `json:"name"`
    Price    float64 `json:"price"`
    Category string  `json:"category"`
    Note     string  `json:"note"`
}
```

```go
func (c *ProductController) Create() {
    var input CreateProductInput
    if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
        c.RespondError(400, "Invalid JSON")
        return
    }
}
```

Named structs are easier to test, document, and reuse.

## JSON Struct Tags

Struct tags control how `json.Unmarshal` maps JSON keys to struct fields.

```go
type Input struct {
    Name      string  `json:"name"`           // JSON key "name"     → Name
    UnitPrice float64 `json:"unit_price"`     // JSON key "unit_price" → UnitPrice
    IsActive  bool    `json:"is_active"`      // JSON key "is_active" → IsActive
}
```

**Without a tag**, Go uses the exact field name. `Name` maps to `"Name"`,
not `"name"`. Most JSON APIs use lowercase or snake_case keys, so tags are
almost always necessary.

```go
// Without tags — JSON key must match field name exactly (case-insensitive match)
type Input struct {
    Name string   // matches "name", "Name", "NAME" — Go is lenient here
}

// With tags — explicit, unambiguous, preferred
type Input struct {
    Name string `json:"name"`
}
```

### The `omitempty` option

`omitempty` has no effect on unmarshaling — it only affects marshaling
(encoding a struct to JSON). Do not use it on input structs for the purpose
of making fields optional. Use pointers instead (shown below).

### The `,string` option — when clients send numbers as strings

Some clients (especially frontend JavaScript) occasionally send numeric
values wrapped in strings: `{"price": "120.50"}` instead of `{"price": 120.50}`.
Normally this causes a type mismatch error. The `,string` tag option tells
`json.Unmarshal` to accept a quoted string and parse it into the numeric type:

```go
type Input struct {
    Price float64 `json:"price,string"` // accepts "120.50" (string) → 120.50 (float64)
    Count int     `json:"count,string"` // accepts "5" (string) → 5 (int)
}
```

**Important caveat:** `,string` only works one way — it expects the value to
always arrive as a quoted string. If the client sends a normal unquoted number
`{"price": 120.50}`, it fails with:

```
json: invalid use of ,string struct tag, trying to unmarshal unquoted value into float64
```

Use this only when you fully control the client and know it always sends numbers
as strings. For clients sending standard JSON types, the plain tag is correct.

## Handling Optional Fields with Pointers

When a field is optional — the client may or may not send it — use a pointer.

```go
type UpdateProductInput struct {
    Name  *string  `json:"name"`   // nil if not sent
    Price *float64 `json:"price"`  // nil if not sent
}
```

```go
var input UpdateProductInput
json.Unmarshal(c.Ctx.Input.RequestBody, &input)

if input.Name != nil {
    // client sent a name — update it
    product.Name = *input.Name
}
if input.Price != nil {
    // client sent a price — update it
    product.Price = *input.Price
}
```

Without pointers, a missing field gets the zero value (`""` for string,
`0` for float64). You cannot tell whether the client sent `0` intentionally
or simply did not send the field at all.

## What `json.Unmarshal` Does and Does Not Do

### What it does

- Decodes valid JSON bytes into a Go struct
- Ignores JSON keys that do not match any struct field (silently)
- Sets unmatched struct fields to their zero value
- Returns an error for malformed JSON (syntax errors, wrong types)

### What it does not do

- Validate that required fields are present
- Check that values are within acceptable ranges
- Verify string formats (email, date, UUID, etc.)

```go
// This succeeds with no error even though "name" is missing from the JSON
var input struct {
    Name  string `json:"name"`
    Email string `json:"email"`
}
json.Unmarshal([]byte(`{"email":"alice@example.com"}`), &input)
// input.Name == ""  ← zero value, no error
```

Validation after unmarshaling is your responsibility.

## Validation After Unmarshaling

Always validate input values after a successful `json.Unmarshal`.
Treat them as untrusted until validated.

```go
func (c *ProductController) Create() {
    var input struct {
        Name     string  `json:"name"`
        Price    float64 `json:"price"`
        Category string  `json:"category"`
        Date     string  `json:"date"`
    }

    // Step 1 — parse
    if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
        c.RespondError(400, "Invalid JSON")
        return
    }

    // Step 2 — validate required fields
    if input.Name == "" {
        c.RespondError(400, "Name is required")
        return
    }
    if input.Price <= 0 {
        c.RespondError(400, "Price must be greater than zero")
        return
    }
    if input.Category == "" {
        c.RespondError(400, "Category is required")
        return
    }

    // Step 3 — validate formats
    validCategories := []string{"Electronics", "Clothing", "Food"}
    if !contains(validCategories, input.Category) {
        c.RespondError(400, "Invalid category")
        return
    }

    // Step 4 — use
    product, err := models.CreateProduct(input.Name, input.Price, input.Category)
    if err != nil {
        c.RespondError(500, "Failed to create product")
        return
    }

    c.RespondJSON(201, true, "Product created", product)
}

func contains(slice []string, value string) bool {
    for _, v := range slice {
        if v == value {
            return true
        }
    }
    return false
}
```

## Reusable Validation — Extract to a Method

When validation logic is repeated across methods, extract it:

```go
type CreateProductInput struct {
    Name     string  `json:"name"`
    Price    float64 `json:"price"`
    Category string  `json:"category"`
}

// Validate checks all fields and returns the first error message found,
// or an empty string if input is valid.
func (i CreateProductInput) Validate() string {
    if i.Name == "" {
        return "Name is required"
    }
    if i.Price <= 0 {
        return "Price must be greater than zero"
    }
    if i.Category == "" {
        return "Category is required"
    }
    return ""
}
```

```go
func (c *ProductController) Create() {
    var input CreateProductInput

    if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
        c.RespondError(400, "Invalid JSON")
        return
    }

    if msg := input.Validate(); msg != "" {
        c.RespondError(400, msg)
        return
    }

    // input is clean — proceed
}
```

This keeps controller methods short and makes validation independently testable.

## Type Mismatches — What `json.Unmarshal` Returns as an Error

If the client sends the wrong type for a field, `json.Unmarshal` returns
an error. Your `400 Invalid JSON` handler catches it:

```go
// Struct expects float64 for "price"
// Client sends: {"name":"Chair","price":"not-a-number"}
//
// json.Unmarshal returns:
// json: cannot unmarshal string into Go struct field .price of type float64
```

```go
if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
    c.RespondError(400, "Invalid JSON")
    return
}
```

The error message from `json.Unmarshal` is technical and internal — do not
send it directly to the client. Always return a clean, human-readable message.

```go
// WRONG — exposes internal details to the client
c.RespondError(400, err.Error())

// CORRECT — clean message for the client
c.RespondError(400, "Invalid JSON")
```

## Empty Body — Handling Missing Request Body

If the client sends a request with no body at all,
`c.Ctx.Input.RequestBody` is an empty `[]byte{}`.
Calling `json.Unmarshal` on an empty slice returns an error.
Your existing error check handles this automatically:

```go
if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
    c.RespondError(400, "Invalid JSON")
    return
}
// This branch is reached for both malformed JSON and missing body
```

If you want a more specific message for an empty body:

```go
if len(c.Ctx.Input.RequestBody) == 0 {
    c.RespondError(400, "Request body is required")
    return
}

if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
    c.RespondError(400, "Invalid JSON")
    return
}
```

## Common Mistakes

### Forgetting `copyrequestbody = true`

```go
// input fields are all zero values, err is nil — silent failure
json.Unmarshal(c.Ctx.Input.RequestBody, &input)
```

Add to `conf/app.conf`:
```ini
copyrequestbody = true
```

---

### Passing the struct by value instead of pointer

```go
// WRONG — Unmarshal cannot modify a copy
json.Unmarshal(c.Ctx.Input.RequestBody, input)

// CORRECT — pass a pointer so Unmarshal can write into the struct
json.Unmarshal(c.Ctx.Input.RequestBody, &input)
```

`json.Unmarshal` requires a pointer. Passing a value compiles cleanly, but
at runtime `json.Unmarshal` uses reflection to check the type immediately
and returns an explicit error:

```
json: Unmarshal(non-pointer main.Input)
```

The struct is not modified and the error is returned — it does not silently
succeed. However, if you are not checking the error return value of
`json.Unmarshal`, this failure goes unnoticed, which is why the error check
is non-negotiable.

---

### Trusting `json.Unmarshal` to validate input

```go
// WRONG — assuming a successful unmarshal means input is valid
if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
    c.RespondError(400, "Invalid JSON")
    return
}
// proceeding without checking if required fields are present
models.CreateProduct(input.Name, input.Price) // input.Name may be ""
```

Always validate after parsing.

---

### Sending the internal error message to the client

```go
// WRONG — leaks implementation details
c.RespondError(400, err.Error())

// CORRECT
c.RespondError(400, "Invalid JSON")
```

## Quick Reference

```go
// conf/app.conf — required
// copyrequestbody = true

// 1. Define input shape
var input struct {
    Name  string  `json:"name"`
    Price float64 `json:"price"`
}
// or a named struct:
// var input CreateProductInput

// 2. Check for empty body (optional but explicit)
if len(c.Ctx.Input.RequestBody) == 0 {
    c.RespondError(400, "Request body is required")
    return
}

// 3. Parse
if err := json.Unmarshal(c.Ctx.Input.RequestBody, &input); err != nil {
    c.RespondError(400, "Invalid JSON")
    return
}

// 4. Validate
if input.Name == "" {
    c.RespondError(400, "Name is required")
    return
}

// 5. Use
result, err := models.Create(input.Name, input.Price)
```

| Situation | Result from `json.Unmarshal` |
|---|---|
| Valid JSON, all fields present | `nil` error, struct populated |
| Valid JSON, field missing | `nil` error, field is zero value |
| Valid JSON, wrong field type | non-nil error |
| Malformed JSON | non-nil error |
| Empty body (`[]byte{}`) | non-nil error |
| Pointer not passed (`input` instead of `&input`) | compiles, but returns `json: Unmarshal(non-pointer ...)` at runtime |
